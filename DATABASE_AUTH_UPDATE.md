# 认证系统数据库化更新文档

## 概述

本次更新将认证系统从使用全局密钥文件改为从数据库中读取每个用户的密钥对进行验证。这提供了更好的用户隔离和安全性。

## 主要变更

### 1. 认证逻辑变更

**原来的方式：**
- 所有用户共享同一对密钥（从配置文件或密钥文件读取）
- 验证时使用全局的 `crypto_auth` 实例

**新的方式：**
- 每个用户在数据库中有独立的密钥对（存储在 `users` 表的 `private_key` 和 `public_key` 字段）
- 验证时根据客户端ID从数据库读取对应用户的密钥，创建临时认证实例进行验证

### 2. 文件修改详情

#### `src/auth/simple_crypto_auth.py`

1. **新增函数：** `_require_database_crypto_auth(f)`
   - 从数据库读取用户密钥进行认证
   - 自动更新用户活跃时间
   - 设置 `request.auth_type = 'database_crypto'`

2. **修改函数：** `require_auth(f)`
   - 现在调用 `_require_database_crypto_auth` 而不是 `_require_crypto_auth`
   - 添加了详细的文档说明

#### `src/app.py`

1. **修改函数：** `init_auth_system()`
   - 添加了数据库认证的说明信息
   - 保留全局 `crypto_auth` 实例初始化以确保兼容性
   - 如果全局密钥配置失败，不会抛出异常，而是继续使用数据库认证

### 3. 新增工具文件

#### `database_auth_example.py`
- 演示如何生成用户密钥对
- 演示如何创建认证令牌
- 提供数据库更新示例

#### `generate_user_keys.py`
- 批量为现有用户生成密钥对
- 查看用户密钥配置状态
- 使用方法：
  ```bash
  python generate_user_keys.py generate  # 生成密钥
  python generate_user_keys.py list      # 查看状态
  ```

## 数据库要求

确保 `users` 表包含以下字段：
- `private_key` (TEXT) - 用户的私钥（PEM格式）
- `public_key` (TEXT) - 用户的公钥（PEM格式）

## 使用方法

### 1. 为用户生成密钥对

```bash
# 自动为所有没有密钥的用户生成密钥对
python generate_user_keys.py generate

# 查看所有用户的密钥配置状态
python generate_user_keys.py list
```

### 2. 客户端认证

客户端需要：
1. 拥有自己的私钥（从数据库获取或管理员提供）
2. 使用私钥生成认证令牌
3. 在 API 请求头中添加：`X-Auth-Token: <base64_encoded_token>`

### 3. 认证流程

1. 客户端发送包含 `client_id`（用户名）的认证令牌
2. 服务端从数据库读取该用户的公钥
3. 使用公钥验证令牌签名
4. 验证通过后更新用户活跃时间

## 兼容性

- 保持了与原有简单API密钥认证的兼容性
- 保留了全局 `crypto_auth` 实例的初始化（用于其他可能的用途）
- 所有现有的 `@require_auth` 装饰器无需修改

## 安全优势

1. **用户隔离：** 每个用户有独立的密钥对，一个用户的密钥泄露不会影响其他用户
2. **动态管理：** 可以随时为用户更新或撤销密钥
3. **审计跟踪：** 自动记录用户活跃时间，便于审计
4. **细粒度控制：** 可以基于用户级别进行访问控制

## 注意事项

1. 首次部署需要为现有用户生成密钥对
2. 客户端需要更新以使用用户特定的私钥
3. 确保数据库连接稳定，认证依赖数据库查询
4. 建议定期备份用户密钥数据

## 故障排除

### 问题：用户认证失败，提示"密钥未配置"
**解决：** 运行 `python generate_user_keys.py generate` 为用户生成密钥

### 问题：认证处理错误
**检查：** 
1. 数据库连接是否正常
2. `users` 表是否存在必要的字段
3. 用户名是否存在于数据库中

### 问题：兼容性问题
**说明：** 全局 `crypto_auth` 实例仍然可用，如有需要可以回退到原来的认证方式
